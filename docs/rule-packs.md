# Rule-pack design

Third-party rule packs can make `agent-config-linter` useful for organization-specific controls, but they also create a code-loading trust boundary. This document defines the safe design target for future work. The current implementation does not load rule packs and MUST NOT execute third-party rule code.

## Non-goals for now

- Dynamic plugin execution is not implemented.
- Rule packs MUST NOT execute Python, shell commands, subprocesses, package installers, network fetches, or arbitrary expressions during discovery or linting.
- Rule packs are not a mechanism for replacing built-in `ACL-*` rules or changing their default behavior.
- Rule packs do not bypass the normal report compatibility, fixture, and documentation requirements.

## Trust boundaries

A rule pack crosses three boundaries: package distribution, local filesystem parsing, and CI policy enforcement. Future implementations should treat rule-pack content as data only, verify manifests before use, and keep evaluation deterministic. Recommended guardrails:

- Parse only a non-executable manifest format such as YAML or JSON.
- Reject unknown executable-looking fields such as commands, entry points, modules, import paths, hooks, or scripts.
- Do not install dependencies declared by a rule pack during linting.
- Require users to opt in to a local path or pinned package artifact before a pack can affect CI.
- Include the rule-pack name and version in reports so baseline and policy decisions remain auditable.

## Manifest schema

A future non-executable manifest should include metadata and declarative rule descriptors only:

```yaml
schema_version: rule-pack/v0
name: org-agent-controls
version: 0.1.0
publisher: Example Security
rules:
  - id: ORG-001
    name: forbid-production-secrets-in-dev-agents
    default_severity: high
    confidence: high
    title: Production secrets exposed to development agent tools
    evidence: Environment declares production credentials next to dangerous tools.
    remediation: Scope secrets to the minimum environment and require approval gates.
    docs: docs/org-001.md
    fixtures:
      risky: fixtures/org-001-risky.yaml
      safe: fixtures/org-001-safe.yaml
```

The manifest intentionally omits executable fields. Match predicates and rule evaluation language need a separate security review before implementation.

## Rule identity and compatibility

Built-in rules keep `ACL-*` IDs. Third-party packs should use a namespace that cannot collide with built-ins, such as `ORG-*` or `VENDOR-*`. Rule identity should be stable across releases and include:

- Machine-readable rule ID and rule name.
- Default severity and confidence.
- Human-readable evidence and remediation.
- Report compatibility notes for added or changed fields.
- A rule-pack version so baseline entries can record which pack produced a finding.

Rule-pack findings should preserve existing report fields, including `schema_version`, `rule_id`, `id`, `severity`, `confidence`, `evidence_paths`, `source_evidence_paths`, and `remediation`.

## Fixtures and documentation

Every accepted rule-pack rule should include at least one risky fixture and one safe fixture. Fixtures should be data files, not generated by code at lint time. Documentation should explain rule intent, false-positive boundaries, and safe suppression examples.

## Future implementation tasks

1. Add manifest parsing tests that reject executable-looking fields.
2. Add a `RulePackManifest` data model without evaluation logic.
3. Add CLI discovery that lists manifest metadata but cannot execute pack rules.
4. Add report fields for rule-pack name/version behind golden tests.
5. Design a sandboxed declarative matcher only after the manifest-only workflow is reviewed.
